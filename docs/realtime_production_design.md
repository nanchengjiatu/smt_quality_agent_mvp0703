# 生产环境实时数据接入设计

状态:2026-07-05 定稿,第一、二阶段已随本文档实施;2026-07-06 第三阶段实施完毕,
§7 两个遗留决策点已拍板(见文末),实时写入表确认为 full_excel 系。

## 1. 问题

MVP 阶段的数据是静态快照(full_excel0625,46,587 行 / 1,659 块板 / 约 37 天)。
真正上产线后,SPI 数据是持续写入的,现有工程有三处扛不住:

1. **每次刷新拖全史**。`run_pipeline` 每轮 `select *` 整表导出(当前 89 MB JSON、
   2.3 s),再全量重算。按实测速率(约 1,300 行 / 45 块板每天),一年后约 50 万行、
   900 MB,刷新时长与内存随历史线性恶化,而 watcher 默认每 5 s 检查一次。
2. **fdate 文本序陷阱(潜伏 bug)**。`fdate` 是 varchar,格式 `2024/1/4 9:58`
   (月/日/时不补零)。变更指纹里的 `max(fdate)` 按文本比较,跨月会错:
   `"2024/9/…" > "2024/11/…"`。当前数据只跨到 1 月所以没炸,生产上跑到 9 月之后
   指纹会卡死在 9 月,新数据不再触发刷新。Python 侧的 `parse_fdate` 是健壮的,
   问题只在 SQL 侧。
3. **重启丢状态**。数据版本号和指纹只在内存里,服务重启后版本归零、指纹为空,
   前端的增量感知和"是否需要刷新"的判断都从头来。

## 2. 关键事实(决定方案走向)

- **所有分析的回溯都是有界的**:precursor 回看 20 块板(`PRECURSOR_LOOKBACK_BOARDS`),
  drilldown 的 pad 序列窗口 300 条、整板上下文 500 条。没有任何模块需要无限历史。
  → 只要滑动窗口 ≥ 最大回溯需求,窗口化重算与全量重算的结论**逐字节一致**。
- **数据速率很低**:平均约 1 行/分钟、45 块板/天。窗口重算一轮只要几秒,
  不需要真正的流式增量状态机(复杂度高、易错、收益为零)。
- 实测:按板取窗(最近 200 块板 = 4,635 行)查询 0.26 s。

## 3. 方案:滑动窗口重算(三层)

### 3.1 变更检测层(watcher,已有,修正)

- 指纹改为 `count(*) + max(解析后的时间戳)`,SQL 侧用正则守卫的
  `to_timestamp(substring(fdate from '…'), 'YYYY/MM/DD HH24:MI')`:
  格式不符的行安全地变 NULL,不会让整条查询失败,也修掉文本序陷阱。
- 指纹仍查全表 count(廉价,50 万行也在几十 ms 量级);百万行以上再考虑
  给生产表加时间戳列 + 索引(见 §6 数据侧需求)。
- 局限(接受):对已有行的 UPDATE 不改变 count 和 max,不会被感知。
  SPI 数据按追加写入假设处理。

### 3.2 数据获取层(核心改动):按板数滑窗

- `run_pipeline` 不再整表拉,改为**最近 N 块板**的行(N = 配置项
  `realtime_window_boards`,默认 3000;0 = 不限,即旧行为):

  ```sql
  with board_times as (
      select barcode, max(<解析时间戳>) as ts from <表> group by barcode
  ), recent_boards as (
      select barcode from board_times order by ts desc nulls last limit N
  )
  select * from <表> where barcode in (select barcode from recent_boards)
  ```

- 按**板数**而不是天数取窗:板数直接对应各分析的回溯语义(20 板 / 500 条),
  且停线数日后重开机不会出现空窗口。
- 一次读、一份快照喂全部三个 stage 的既有一致性保证原样保留,只是快照
  范围从"全史"变为"窗口"。
- 默认 3000 板 ≈ 66 天产量,远大于最大回溯需求(约 20 板),也大于当前
  全部数据(1,659 板)——**上线当天输出与全量完全一致,无语义跳变**;
  数据长过窗口后,旧板自然滚出实时视图。

### 3.3 视图/语义层

- 刷新报告与 `/api/live` 暴露 `window_boards` 与实际加载板数,前端可据此
  在页面标注统计口径("最近 N 块板")。UI 标注在数据真正超过窗口前不改
  (当前窗口覆盖全量,口径没变,提前标注反而误导)。
- 历史全量分析改为**按需**:`POST /api/refresh` 带 `{"window_boards": 0}`
  即可跑一次全量,不再每 5 秒重算全史。
- 触发器跨轮稳定性:重算会重新生成 triggers,窗口滚动后早于窗口的触发
  自然过期——这是实时视图的正确行为。若前端将来要做"已读/确认"状态,
  需要确定性 trigger_id(起止板 + pad 派生),列入第三阶段。

### 3.4 状态持久化

- 每次 pipeline 成功后把 `{version, updated_at, fingerprint}` 落盘到
  `output/live_state.json`,启动时回读。重启后版本号续增、指纹延续,
  `--no-refresh-on-start` 模式下若数据没变不会触发多余的重算。

## 4. 不做(本期非目标)

- 真正的增量/流式计算(每板增量更新分析状态)——速率不支撑其复杂度。
- 本地镜像库(SQLite 增量摄取)——只有当生产库不允许被反复查询、或窗口
  查询变贵时才需要;届时摄取层单独加,分析层不用动。
- SSE/WebSocket 推送——前端轮询 `/api/live` 在这个速率下足够。

## 5. 阶段划分

| 阶段 | 内容 | 状态 |
|---|---|---|
| 1 | 指纹修正(时间戳解析)+ 状态落盘恢复 | 已实施(2026-07-05) |
| 2 | 按板数滑窗加载,全量分析转按需,窗口口径进报告 | 已实施(2026-07-05) |
| 3 | 前端窗口口径标注、确定性 trigger_id、新触发提醒 | 已实施(2026-07-06) |
| 数据侧 | 见 §6 | 部分确认,余项待推进 |

阶段 3 实施要点:

- **确定性 trigger_id**:`TRG-<sha1(model|pad|起始板barcode|起始时间)[:8]>`,
  由触发串的起始板身份派生而非枚举序号,窗口滚动/重算后同一触发 id 不变,
  前端已读/确认状态可跨重算延续;展示序号另存 `trigger_no`。
- **窗口口径标注**:dashboard_summary 新增 `scope`(window_boards/loaded_boards),
  前端仅在 `loaded_boards ≥ window_boards`(数据真正超窗)时在指标区下方标注
  "最近 N 块板"口径,超窗前不标以免误导。
- **新触发提醒**:自动更新时对比确定性 trigger_id 集合,出现新 id 弹出
  "新三板连发触发"提示(优先级高于普通"新异常"提示)。

## 7. 遗留决策点(2026-07-06 已全部拍板)

1. **窗口大小:维持默认 3000 板**(≈66 天)。如需调整改
   `config/datasource.json` 的 `realtime_window_boards`,无需改码。
2. **dashboard 累计数字:双口径**。主数字维持窗口内口径;真累计不做本地累计器,
   而是每轮刷新时对全表跑一次纯聚合(`cumulative_stats_query`:行数/板数/
   NG 行数/NG 板数/起止时间,NG 口径与 `is_ng` 一致),作为 `cumulative`
   随 dashboard_summary 下发,前端在指标区下方作副标注。全表聚合不拉明细,
   50 万行量级仍是几十 ms;聚合失败不拖垮分析 stage(副标注如实消失)。
