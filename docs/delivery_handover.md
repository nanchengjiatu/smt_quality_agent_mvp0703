## 一、这套系统是什么

一句话：**盯着 SPI（锡膏检测）数据流的质量分析 Agent**——异常发生时自动下钻找根因、
给处置建议；异常发生前用漂移监控提前预警；全部结论可审计、可追问。

能力清单（对应前端六个页签）：

| 能力 | 页签 | 干什么 |
|---|---|---|
| 实时异常 | 实时异常 | 逐行判定 NG 类型/风险等级/异常模式，生成质量案例 |
| 事前预警 | 事前预警 | 每个焊盘的 EWMA 漂移监控 + 健康矩阵，NG 之前报警 |
| 质量案例 | 质量案例 | 中高风险异常沉淀为案例，跟踪复测与关闭 |
| 统计看板 | 统计看板 | 汇总指标 + TOP 排名（窗口口径为主、全表累计为副） |
| 事件与根因 | 事件与根因分析 | NG 板聚类成事件、前兆回溯、参数排除、工艺维度体检 |
| 三板连发下钻 | （从实时异常/事件进入） | 同一焊盘连续 ≥3 板 NG 的完整根因分析工作台 + 对话 |
| 知识库 | 机理手册 | 13 个失效机理手册 + 维护视图（词表/决策流程/处置阶梯） |

工程特征（讲给客户的技术负责人）：

- **零第三方依赖**：纯 Python 标准库 + 前端原生 JS，部署 = 一个 Python 进程；
  数据库访问走 `psql` 子进程，LLM 调用走 `urllib`。
- **约 7300 行 Python（15 个模块）+ 3 个前端文件，190 个单元测试。**
- 所有分析每轮**从同一份数据快照全量重算**（幂等），没有增量状态机——
  数据速率（约 45 板/天）远不需要流式计算的复杂度。

## 二、总体架构与数据流

```
PostgreSQL (l780db)                     serve.py (唯一进程,默认 :8502)
┌──────────────────┐   指纹轮询(5s)   ┌──────────────────────────────┐
│ full_excel 系表   │◄────────────────│ Watcher 线程                  │
│ (生产实时写入表)   │                 │   指纹=行数+最新时间戳          │
└──────────────────┘   变化→重算      │   变化才触发 run_pipeline      │
        │                            ├──────────────────────────────┤
        │ 滑动窗口取数(最近3000板)      │ run_pipeline (pipeline.py)    │
        ▼                            │  一次读取 → 四个 stage 共享快照 │
   一份行快照 ──┬─► anomaly_cases ────►│ output/abnormal_results.json  │
              ├─► param_analysis ───►│ output/param_analysis.json    │
              ├─► drilldown ────────►│ output/drilldown.json         │
              └─► early_warning ────►│ output/early_warning.json     │
                                     ├──────────────────────────────┤
                                     │ HTTP 服务                     │
                                     │  /web/* 静态前端              │
                                     │  /output/* 分析结果 JSON      │
                                     │  /api/*  动态接口(见第八节)    │
                                     └──────────────────────────────┘
                                                  ▲
                                     浏览器：每 4s 轮询 /api/live，
                                     版本号变了才重新拉数据
```

要点讲解：

1. **单快照一致性**：四个 stage 消费同一次数据库读取的结果，所以实时异常、
   事件、下钻、预警之间**永远不会互相矛盾**（不存在"两次查询之间来了新数据"）。
2. **滑动窗口**：每轮只加载最近 N 块板（配置 `realtime_window_boards`，默认
   3000 ≈ 66 天产量；0=全表）。所有分析的回溯需求都远小于窗口（最大 20 板 /
   500 行），因此窗口重算与全量重算在重叠范围内**逐字节一致**（已验证）。
   需要历史全量时 `POST /api/refresh` 带 `{"window_boards": 0}` 按需跑。
3. **变更感知**：watcher 的指纹是 `count(*) + max(解析后的时间戳)`。注意
   `fdate` 是不补零的文本（`2024/1/4 9:58`），直接 `max(fdate)` 文本序跨月会错，
   所以 SQL 侧一律用 `pipeline.fdate_ts_expr()`（正则守卫的 to_timestamp）。
4. **重启不丢状态**：`output/live_state.json` 落盘版本号/指纹/窗口；重启后
   版本续增，前端无感。
5. **确定性 ID**：三板连发触发 `TRG-<hash>`、漂移预警 `WRN-<hash>` 都由
   "起始板身份"哈希派生——窗口滚动、服务重启、重算之后 **ID 不变**，
   已读/确认状态因此能跨重算延续。

## 三、数据与口径（先讲清楚，后面全靠它）

**数据源**：`l780db.public.full_excel0625`（配置 `tables.full_spi`，客户产线
以实际的 full_excel 系实时写入表为准）。每行 = 一块板上一个焊盘的一次 SPI 测量，
约 87 列：板条码/时间/机种/焊盘名 + 三个测量偏差 + 印刷参数三元组
（`<参数>_plan` 计划值 / `<参数>` 实际值 / `diff_<参数>` 差值）。

**五个必须讲对的口径**（历史上都踩过坑）：

1. **`Comp_avdp/aadp/ahdp` 是无符号偏差幅度**（体积/面积/高度），全表最小值 0。
   多锡和少锡的 avdp 可能一样大——**缺陷方向只在 `Comp_errName` 标签里**。
   因此所有监控都是"幅度爬升=变坏"的单边逻辑，机理签名表达的是"哪些指标劣化"
   而不是物理升降方向。
2. **NG 判定**：`Comp_errName` 非空且大写后 ≠ "PASS"（`param_correlation.is_ng`）。
3. **首检 vs 复测**：同一板条码在更晚时刻再次出现 = 复测。复测**不算生产板**：
   不参与连发判定、不进基线，但保留在趋势图和恢复判定里（复测 PASS 是
   "已恢复"的证据）。
4. **fdate 是文本**：Python 侧统一 `parse_fdate()`，SQL 侧统一 `fdate_ts_expr()`。
   任何新查询直接比较 fdate 文本都是 bug。
5. **`*_plan` 是伪设定值陷阱**：部分"计划值"列逐板都在变（跟随实际值），
   不能当真正的程序设定值用；参数变更事件检测已对此做了处理。

## 四、四条分析流水线（`pipeline.py` 编排，故障互相隔离）

任何一个 stage 失败只影响自己的 JSON 文件，其余照常产出；前端对失败的
stage 显示诚实的错误态而不是空白。

### 4.1 anomaly_cases —— 实时异常与质量案例（`rules_engine.py`）

首检行逐行过规则引擎：判 NG 类型（多锡/少锡）、风险等级（高/中/低）、
异常模式（孤立/同板多点/连续同点……），中高风险沉淀为质量案例。
输出 `abnormal_results.json` / `quality_cases.json` / `dashboard_summary.json`
（含 `scope` 窗口口径 + `cumulative` 全表真累计——真累计来自一条纯聚合 SQL，
不拉明细，失败不拖垮 stage）/ `dashboard_top.json`。

### 4.2 param_analysis —— 事件与根因分析（`param_correlation.py` + `process_dimensions.py`）

- **事件聚类**：同机种 NG 板按 30 分钟间隔聚成"事件"；
- **前兆回溯**：事件前 20 块正常板的板均偏差，斜率 ≥0.5%/板 且末段连升 ≥3 板
  判"渐变型（有前兆）"，否则"突发型"；
- **参数排除法**：事件时段的参数实际-计划偏差与基线对比，给出
  "参数无偏离/某参数偏离"的核查结论；
- **复测有效率**：复测 PASS 占比；
- **工艺维度体检**（P1）：清洗周期效应/停机首板效应/印刷方向差异三张体检卡，
  每轮重算，如实报告四态之一（存在效应/无明显效应/样本不足/数据未采集），
  判效应门槛 = max(1pp, 0.5×板均噪声σ)。当前数据三个维度均无效应——
  这是**如实结论**，数据变了会自动点亮。

### 4.3 drilldown —— 三板连发下钻（`drilldown.py`，系统的核心深度）

**触发**：同一"机种×焊盘"连续 ≥3 块生产板 NG 且中间无 PASS（严格口径，
复测不计入）。触发 ID 确定性派生。

**围绕触发自动建包**（详见第六节的完整案例走读）：
- 焊盘自身序列窗口（前后各 300 条）+ 全量 SPI 上下文窗口（前后各 500 条，
  **含所有焊盘**——范围判定必须看别的 Pad）；
- 观测层判定：范围（单Pad孤立/同元件多Pad/局部区域/整板同向/疑似SPI假异常）、
  趋势（突变/渐变）、恢复、周期性、参数核查、三指标签名（判界 max(3σ,10pp)）、
  擦网频率参照；
- 决策四步（第六节）产出根因候选（前3）+ 处置分级 + 复判计划；
- **`analysis_contract` 是唯一权威结论载荷**（版本 analysis-contract-v4）：
  前端展示、对话回答、诊断轨迹全部以它为准，不存在第二份结论。

### 4.4 early_warning —— 事前预警（`early_warning.py`，P0）

- **监控**：92 个焊盘 + 1 条整板均值序列，各自做三指标 EWMA
  （λ=0.2 / L=4.0，回测标定，配置可调）；基线 = 滚动 200 块板的 PASS 值，
  **报警期间冻结**（漂移不能拉高自己的控制限），预热 30 板；
- **分级**：L1 观察（越限1板）→ L2 预警（连续≥3板或双指标共振）→
  L3 高危（L2 且末段连升≥3，或 avdp 进入历史 NG 观测带）；
- **消费方式**（评审拍板）：只有**活动中的 L3** 弹页面提示（约 1.5 条/天）；
  L1/L2 沉淀进 Pad 健康矩阵（按漂移裕度 margin 着色）；
- **台阶式新常态**：越限持续 ≥100 板不回落的活动告警转入"待确认新常态"桶，
  工程师点"接受为新基线"→ ack 按确定性 ID 落盘 `output/warning_acks.json`
  → 该焊盘从迁移点重启监控（跨重算生效）；
- **margin**：距历史 NG 最低观测水位还有几个 3σ。**注意这不是真 Cpk**——
  SPI 判定阈值未拿到，页面如实标注；阈值到位后同字段换真 Cpk。

## 五、知识层——机理手册与规则（`ontology.py` + `knowledge_base.py`）

**分层原则**（机理手册页底部"维护视图"里有同样的说明）：

| 层 | 回答什么 | 内容 | 说错了怪谁 |
|---|---|---|---|
| 观测层 | 数据里发生了什么 | 三个判定轴（空间×时间×有效性，11 个取值）+ 25 种证据 | 计算逻辑 bug → 改代码 |
| 机理层 | 为什么会这样 | **13 个失效机理**（根因词表唯一权威） | 知识错 → 改机理卡 |
| 决策层 | 怎么判怎么办 | 10 条决策规则 + 6 级处置阶梯 + 置信度算式 | 策略松紧 → 调阈值/顺序 |
| 实体层 | 落在哪个物理对象 | 工序/部位/物料骨架（薄，待 MES） | — |

**机理是单源**：诊断候选、预警提名、chat 回答、事件归因里出现的每个根因
说法都引用同一份机理定义。每张机理卡四个可执行字段：

- **先验**（confidence_base）：核对证据前的初始把握度（如范围先验 0.7、
  投影兜底 0.45）——决策第 2 步的起点分数；
- **签名**（signature）：该机理发作时三指标各自的表现（如残锡转印
  "体积↑ 面积↑ 高度平/↑"）——决策第 3 步吻合 ×1.2 / 方向冲突 ×0.7；
- **典型范围**（typical_spatial）：驱动候选投影和空间典型性 ×1.1/×0.9；
- **动作**（action）：确认后现场干的第一件事，直接进处置建议。

**改知识的流程**：改 `ontology.py` 里的机理定义 → 跑测试（守护测试锚住
词表/绑定/口径）→ 版本串升级（当前 spi-printing-v5 / rule-catalog-v6）。
全系统自动跟随，无需逐处同步。

**置信度模型**（`/api/rules` 的 `confidence_model` 块单源暴露，页面算式卡
自动跟随常量）：

```
最终把握 = 先验 × 签名(吻合1.2/冲突0.7) × 空间典型性(1.1/0.9)
         × 擦网对齐(1.15) × 跨机种基线(0.8)，封顶 0.95
分档：≥0.75 高 / ≥0.5 中 / <0.5 低（分档是推导的，不单独维护）
```

## 六、决策流程完整走读（拿真实案例讲）

给客户演示时用这个真实案例：**2023/12/6 上午，C1754_2 连续 3 块板多锡**
（TRG-d1489186）。

1. **数据到达**：watcher 发现指纹变化 → 拉最近 3000 板快照 → 识别复测行；
2. **触发判定**：C1754_2 在 10:30/10:33/10:36 连续 3 块生产板 Over Volume，
   中间无 PASS → 生成触发；
3. **观测层算事实**（只描述不解释）：
   - 范围：兄弟 Pad 和整板上下文都干净 → **单Pad孤立异常（高）**；
   - 趋势：触发前基线稳定 31.5%、斜率 -0.55%/板 → **突变型**；
   - 恢复：触发后第 1 块板即消失 → **已恢复**，且恢复前有 xbackward/xforward
     程序设定变更（记录在案）；
   - 签名：三指标何者劣化，按 max(3σ,10pp) 判界；SPI 真伪：偏差与标签相符。
4. **决策四步**（全程记入诊断轨迹）：
   - **这个 NG 是真的吗？** 是 → 不走假异常口径；
   - **可能是哪里出的问题？** 提名三条嫌疑：残锡转印（范围先验 0.7）、
     参数漂移（恢复前有设定变更，先验 0.85）、离散制程变化（0.6）；
   - **把握有多大？** 残锡转印签名吻合 ×1.2、单Pad 是其典型范围 ×1.1
     → **0.924 高**；参数漂移的典型范围是整板/局部而这次是单Pad ×0.9 → 0.765；
   - **现场先干什么？** 排序取前3 → 处置 **P2 按首要根因执行现场确认**，
     动作来自机理卡："清洁钢网底面并复测下一块板，核对擦网周期"+ 复判标准。
5. **到人眼前**：实时异常行挂 🔴 徽标、新触发 toast、下钻工作台
   （趋势图/对比/结论/诊断轨迹折叠区——每条规则命中/未命中、置信算式、
   落选原因全在）、对话区可追问（第七节）。

强调给客户的两点：**证据冲突不和稀泥**（本例"参数变更时序巧合"与"签名指向
残锡"两条证据都列出、按证据强度排序）；**每一步可审计**（诊断轨迹就是
决策流程对这一次触发的逐条求值记录）。

## 七、前端页面导览（`web/` 三个文件，原生 JS 无框架）

- **实时异常**：逐行异常 + 过滤/搜索；三板连发行带 🔴 下钻入口；
- **事前预警**：漂移告警卡（实测/EWMA/控制限三线图 + 机理提名）、
  待确认新常态（"接受为新基线"按钮）、92 Pad 健康矩阵（margin 着色排序）；
- **质量案例 / 统计看板**：案例跟踪与汇总指标（指标区下方脚注 =
  窗口口径标注[数据超窗才显示] + 数据源全表累计）；
- **事件与根因分析**：事件总览、下钻决策卡、**工艺维度体检**三卡、事件列表；
- **机理手册**：13 张机理卡按缺陷方向分组（多锡/少锡/双向/兜底），每卡
  "什么现象→怎么确认→确认后怎么办"；底部折叠"维护视图"（三轴词表、
  关系全景图、决策四步流程+置信度算式卡、处置阶梯）；
- **下钻工作台**（全屏 overlay）：主趋势图（基线带/触发区/参数事件线/
  参数曲线叠加双轴）、兄弟 Pad 对比/热力图、结论与证据、诊断轨迹、
  **对话分析**——已配置 LLM 时由所选大模型基于分析契约+机理目录回答
  （每条回答标注来源），未配置/失败自动回退离线规则问答；
- **顶栏**：实时徽标（连接状态）、机理手册、LLM 配置（六家提供商可选）、
  数据源配置、手动刷新。

自动更新提示优先级：新三板连发触发 > 新 L3 漂移预警 > 新异常 > 普通更新。

## 十、部署与运维

```bash
python3 serve.py                      # 跑 pipeline + watcher + 服务 :8502
python3 serve.py --port 8080 --watch-interval 10
python3 serve.py --no-watch           # 关自动更新
python3 serve.py --no-refresh-on-start  # 启动不重算(数据没变时秒起)
python3 serve.py --database otherdb   # 覆盖库名
```

- 依赖：Python 3.10+、psql 客户端可连库。无 pip install。
- 重启行为：`live_state.json` 保证版本续增、指纹延续、不空刷。
- 需要持久化备份的文件：`config/*.json`（凭据/参数）、
  `output/warning_acks.json`（已接受的新基线）。其余 output/*.json 可重算。
- 网络面：建议防火墙只放行服务端口；服务本身**无认证**，定位是工厂内网工具。
- 生产建议（交付时明说）：screen/systemd 托管 + 存活监控；数据量到百万行级
  时给生产表加真 timestamp 列或自增 id 并建索引（见
  `docs/realtime_production_design.md` §6）。

## 十二、测试体系（190 个用例的守护对象）

- **口径守护**：NG 判定、首检/复测、fdate 解析、无符号幅度——错了整个
  系统跟着错的地方；
- **知识守护**：机理词表完整性、规则绑定、投影行为、置信度算式（数值变了
  测试先叫）；
- **确定性守护**：trigger_id/warning_id 跨窗口滑动不变；
- **行为测试**：合成数据验证触发检测、EWMA 报警/不报警/基线冻结/新常态、
  体检四态、LLM 三协议请求构造与回退（**零外呼**，网络层可注入）。

改任何东西后：`python3 -m unittest discover tests`。测试挂了先看它守护的
口径是不是被无意打破。

## 十三、如何修改与扩展（三层故障定位）

客户接手后遇到"系统说得不对"，先分类再动手：

1. **描述与实际不符**（如明明 3 个 Pad 异常却判"整板同向"）→ 计算逻辑，
   改 `drilldown.py`/`early_warning.py` 的判定函数与阈值常量；
2. **根因说法不符合产线经验**（如某机理的签名/动作不对）→ 知识，改
   `ontology.py` 机理定义，跑测试，版本串升级；
3. **太松/太紧**（误报多、处置激进）→ 策略，改 `knowledge_base.py` 的
   先验/乘数/处置阶梯，或 `config` 里的 EWMA 参数——知识与代码都不用动。

常见扩展点：新机种接入（重跑 `early_warning.backtest_grid` 标定 λ/L）；
新增机理（ontology.py 加卡 + 守护测试）；新增数据列到位（把对应
evidence 的 availability 从 not_collected 改掉，相关机理自动证据点亮）。

## 十四、已知边界（交付时必须如实告知）

1. **所有阈值/先验标定自单机种单机台 37 天数据**（12 条 NG）。换机种/换线
   需重标定（回测 harness 已具备）；
2. **13 个机理的工艺内容为领域先验，尚未经产线专家逐条评审**——根因建议
   在评审完成前应作"参考"使用；
3. **数据侧缺口**：SPI 判定阈值未接入（margin 非真 Cpk）、PrintDirection/
   擦网时刻/温湿度未采集（对应机理无法自动验证）、fdate 为文本（百万行级
   前需真时间戳+索引）；
4. **LLM 对话是数据出厂行为**：启用后触发上下文（机种/缺陷/参数）发送至
   所选大模型服务商，需客户数据治理拍板；不启用则纯离线运行；
5. **服务无认证**，安全边界依赖网络隔离；
6. 工艺维度体检当前三卡为"无效应/样本不足/未采集"是**当前数据的如实结论**，
   不是功能未完成。

## 术语表

| 术语 | 含义 |
|---|---|
| 首检/复测 | 板的第一次检测 / 同板条码更晚时刻的再次检测（不算生产板） |
| 三板连发 | 同一焊盘连续 ≥3 块生产板 NG 且中间无 PASS（严格口径） |
| 三指标 | 体积/面积/高度偏差幅度（Comp_avdp/aadp/ahdp，无符号） |
| 签名 | 机理发作时三指标的预期表现模式 |
| 先验/把握度 | 核对证据前/后的根因置信度（≥0.75 高 / ≥0.5 中） |
| 分析契约 | analysis_contract，一次触发的唯一权威结论载荷 |
| 诊断轨迹 | 决策四步对一次触发的逐条求值记录（审计链） |
| EWMA | 指数加权移动平均，漂移监控的核心统计量 |
| margin | 漂移裕度：距历史 NG 最低观测水位的 3σ 倍数（非真 Cpk） |
| 新常态 | 台阶式均值迁移后不回落的水平，需人工"接受为新基线" |
| 机理 | FailureMechanism，根因词表的唯一权威（13 个） |
